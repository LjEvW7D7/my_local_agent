# Установка ИИ-ментора «Кот»

Инструкция разворачивает проект с нуля: локальную LLM (Ollama), Python-окружение,
доступ к Google Calendar и запуск консольной или веб-версии агента.

Предполагается Windows (напоминания через `ctypes.windll` работают только на
ней) — в конце есть раздел про Linux/macOS, где эта часть просто отключается.

---

## 0. Что потребуется

- Python  3.10+
- ~10 ГБ свободного места на диске под модель Ollama.
- Аккаунт Google (для доступа к календарю).
- Windows 10/11 — если нужны всплывающие уведомления; иначе любая ОС.

---

## 1. Установка Python-зависимостей

1. Создайте папку проекта и положите туда `mentor_core.py`, `app.py`,
   `index.html` (в подпапку `templates/`, см. шаг 5).

2. Создайте виртуальное окружение:

   ```bash
   python -m venv venv
   ```

   Активация:
   - Windows: `venv\Scripts\activate`
   - Linux/macOS: `source venv/bin/activate`

3. Установите библиотеки:

   ```bash
   pip install flask requests playwright playwright-stealth google-auth google-auth-oauthlib google-api-python-client
   ```

   Пояснение, зачем что нужно:
   - `flask` — веб-интерфейс (`app.py`).
   - `requests` — запросы к локальному Ollama.
   - `playwright` + `playwright-stealth` — инструмент `visit_url` (скрытый браузер).
   - `google-auth`, `google-auth-oauthlib`, `google-api-python-client` — доступ к Google Calendar.

4. Установите браузерный движок для Playwright (обязательный отдельный шаг,
   без него `visit_url` не заработает):

   ```bash
   playwright install chromium
   ```

---

## 2. Установка и запуск Ollama (локальная модель)

1. Скачайте и установите Ollama: https://ollama.com/download
   (Windows/macOS/Linux — есть нативные установщики).

2. Запустите сервер (обычно стартует автоматически как служба; если нет):

   ```bash
   ollama serve
   ```

   По умолчанию слушает `http://localhost:11434` — именно этот адрес
   зашит в `ask_ollama_chat()` в `mentor_core.py`.

3. Скачайте модель, которую использует агент:

   ```bash
   ollama pull qwen2.5-coder:14b
   ```

   Это модель на 14 млрд параметров — нужно 8+ ГБ VRAM/RAM для комфортной
   работы. Если железо слабее, можно взять более лёгкую модель (например,
   `qwen2.5-coder:7b`) — тогда в `mentor_core.py` поменяйте:

   ```python
   MODEL_NAME = 'qwen2.5-coder:14b'   # -> замените на нужную модель
   ```

4. Проверьте, что модель отвечает:

   ```bash
   ollama run qwen2.5-coder:14b "привет"
   ```

## 3. Настройка доступа к Google Calendar

Проект использует OAuth Desktop-приложение. Нужно создать свой собственный
`credentials.json` в Google Cloud Console — **не используйте чужой файл
credentials.json/token.json**, если он у вас откуда-то есть: это чужие
рабочие ключи доступа, ими нельзя пользоваться, и их нужно считать
скомпрометированными.

1. Зайдите в [Google Cloud Console](https://console.cloud.google.com/).
2. Создайте новый проект (или выберите существующий).
3. В меню слева: **APIs & Services → Library** → найдите **Google Calendar API**
   → нажмите **Enable**.
4. **APIs & Services → OAuth consent screen**:
   - тип User Type — **External** (если это личный аккаунт, не Workspace);
   - заполните название приложения, email;
   - на шаге Scopes ничего добавлять не обязательно — авторизация сама
     запросит нужный scope при первом запуске;
   - добавьте свой email в **Test users**, если приложение останется в
     статусе "Testing".
5. **APIs & Services → Credentials → Create Credentials → OAuth client ID**:
   - Application type: **Desktop app**;
   - задайте имя, нажмите **Create**.
6. Скачайте JSON (кнопка **Download JSON**) и положите его в папку проекта
   под именем `credentials.json` — именно это имя ожидает
   `get_calendar_service()`.

Файл `token.json` создавать не нужно — он появится автоматически при первом
запуске: откроется браузер, попросит войти в Google-аккаунт и разрешить
доступ к календарю (`scope = calendar`), после чего токен сохранится рядом
с проектом.

> ⚠️ **Важно:** `credentials.json` и особенно `token.json` содержат секреты
> уровня пароля. Никогда не публикуйте их в открытых репозиториях, чатах или
> скриншотах. Добавьте оба файла в `.gitignore`.

## 4. Структура папок проекта

Разложите файлы так:

```
project/
├── venv/                      # виртуальное окружение (шаг 1)
├── mentor_core.py              # ядро агента
├── app.py                      # веб-версия (Flask)
├── templates/
│   └── index.html              # веб-интерфейс (Flask ищет шаблоны в templates/)
├── workspace/                  # создастся автоматически — песочница для read/write_file
├── credentials.json             # ваш собственный OAuth-клиент (шаг 3)
├── token.json                   # создастся автоматически после первой авторизации
├── chat_history.json            # создастся автоматически при первом сообщении
├── tasks.json                   # создастся автоматически при первой задаче
└── progress.json                # создастся автоматически при первом update_progress
```

Важный нюанс: `app.py` вызывает `render_template('index.html')` — Flask по
умолчанию ищет файлы шаблонов в подпапке **`templates/`**, поэтому
`index.html` нужно положить именно туда, а не в корень проекта.

## 5. Первый запуск

### Вариант А — консольная версия

```bash
python mentor_core.py
```

При первом запуске откроется браузер для авторизации Google (шаг 3).
После входа появится приглашение `Вы:` — можно писать сообщения.
Выход — команда `выход`, `exit` или `quit`.

### Вариант Б — веб-версия

```bash
python app.py
```

- Откройте в браузере: **http://127.0.0.1:5000**
- Первый запрос, требующий календарь, точно так же откроет окно авторизации
  Google (если `token.json` ещё нет).
- Чат, боковая панель с задачами/событиями/прогрессом и автообновление раз
  в минуту — всё работает через тот же `mentor_core.py`.

## 6. Проверка, что всё работает

- **Ollama:** `ollama ps` должен показывать загруженную модель после первого сообщения.
- **Календарь:** попросите агента "покажи мои ближайшие события" — должен
  прийти список без ошибок авторизации.
- **Файлы:** "запиши файл test.txt с текстом привет" → проверьте, что файл
  появился в `workspace/test.txt`.
- **Браузер:** "открой сайт example.com" → должен прийти вытащенный текст страницы.
- **Задачи/прогресс:** "напомни мне позвонить маме завтра в 18:00" → должна
  автоматически создаться задача (агент распознаёт неявные намерения).

## 7. Уведомления (только Windows)

`calendar_reminder_worker` и `task_reminder_worker` шлют всплывающие окна
через `ctypes.windll.user32.MessageBoxW` — это Windows-only API.

- На **Windows** — работает из коробки, ничего дополнительно делать не нужно.
- На **Linux/macOS** — эти потоки будут падать с исключением на каждой
  минутной итерации (ошибка перехватывается, приложение не упадёт, но
  уведомлений не будет). Если нужны уведомления на этих системах, замените
  `send_windows_notification` на кроссплатформенный вариант, например:

  ```python
  from plyer import notification

  def send_windows_notification(title, message):
      notification.notify(title=title, message=message, timeout=10)
  ```

  (потребуется `pip install plyer`).

## 8. Типичные ошибки при установке

| Симптом | Причина | Решение |
|---|---|---|
| `Ошибка подключения к Ollama: сервер недоступен` | Ollama не запущена | выполнить `ollama serve` или проверить службу |
| `Ошибка подключения к Ollama: превышено время ожидания` | Модель грузится на CPU и отвечает дольше 10 минут | подождать, использовать GPU или более лёгкую модель |
| Открывается окно логина Google при каждом запуске | `token.json` не сохраняется (нет прав на запись в папку) | запускать не из системной/защищённой директории |
| `Invalid isoformat string` при создании события | Модель подставила в JSON плейсхолдер вместо реальной даты | переформулируйте запрос точнее ("завтра в 15:00"), проверьте `chat_history.json` на мусорные записи |
| `net::ERR_NAME_NOT_RESOLVED` при `visit_url` | Не установлен браузер Playwright | выполнить `playwright install chromium` |
| Веб-версия не находит `index.html` | Файл лежит не в `templates/` | переместить файл в `templates/index.html` |

## 9. Обновление и обслуживание

- Смена модели Ollama — правка `MODEL_NAME` в `mentor_core.py`.
- Сброс обучения "с нуля" — удалить `progress.json` (пересоздастся пустым).
- Полный сброс памяти чата — удалить `chat_history.json`.
- Если `credentials.json`/`token.json` когда-либо "утекли" наружу (например,
  попали в переписку, скриншот, публичный репозиторий) — обязательно зайдите
  в Google Cloud Console → Credentials и пересоздайте/сбросьте client secret,
  затем удалите `token.json`, чтобы приложение переавторизовалось с чистого листа.
